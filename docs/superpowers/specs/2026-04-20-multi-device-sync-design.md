# Multi-Device Account & Sync — Design

**Date:** 2026-04-20
**Status:** Design approved, pending implementation plan
**Scope:** Cross-device identity, state sync, Spotify-Connect-style remote control

## 1. Goals

Enable a single GOAMP user to:

- Use GOAMP on multiple devices (desktop Mac/Win/Linux + future mobile) with shared state (playlists, likes, history, settings, taste profile).
- Control an active playback session from any other device (play/pause/seek/queue/volume — full control).
- Transfer an active session between devices ("continue here") with position preserved.
- Recover all data on a brand-new device via a seed phrase.
- Revoke a lost/stolen device.

All without compromising GOAMP's P2P-first philosophy, privacy, and RU-jurisdiction compliance (no PII on servers).

## 2. Non-Goals (out of scope for this spec)

- Multi-active / group listening ("party mode", Mirror, tick-level sync across devices) — deferred to a future spec.
- Email/password login, OAuth providers, magic-links — deferred; key-based identity only for v1.
- P2P cache/audio content sync — tracked as a separate feature (manual "sync from device X" button exists at feature level, internal sync of binary caches is a separate spec).
- Mobile client implementation — design MUST be mobile-compatible, but mobile UI is deferred per the Phase 1 focus in `project_goamp_decisions`.

## 3. Key Decisions (resolved during brainstorming)

| Topic | Decision |
|---|---|
| Account model | Key-based identity (Nostr-style). Master ed25519 key = account. |
| Infrastructure | Hybrid P2P + optional relay, with cascade: direct libp2p → community relay → official relay. |
| Sync scope | Full Spotify-Connect-style: state sync + remote control + session takeover. |
| Pairing UX | QR pairing between devices as primary flow; BIP39 seed phrase (12 words) as mandatory recovery. No email, no password. |
| Session semantics | Single active device. Explicit takeover with confirmation dialog. `continue-from-position` on takeover. |
| Volume | Local per-device (not synced). |
| History sync | Synced by default, with 3-level privacy switches (global / per-device / incognito session). |
| Offline conflicts | Automerge CRDT for user state; last-write-wins for session state. |
| Snapshot cadence | Realtime deltas via GossipSub; snapshot push every 5 min if changed + immediate on significant events. |
| Device security | Master key + per-device sub-keys. Signed device manifest. Revoke = new manifest version. |
| State blob encryption | ChaCha20-Poly1305, key = HKDF-SHA256(master, "goamp-state-v1"). |

## 4. Architecture Overview

```
┌─────────────────────────────────────────────────────┐
│ Tauri frontend (UI: devices panel, pairing, remote) │
├─────────────────────────────────────────────────────┤
│ goamp-node (Go sidecar)                             │
│  ├── identity/       ed25519 master + sub-keys      │
│  ├── sync/           CRDT user state (Automerge)    │
│  ├── session/        active-device state machine    │
│  ├── remote/         realtime command bus           │
│  └── transport/      libp2p direct + relay client   │
├─────────────────────────────────────────────────────┤
│ libp2p P2P network                                  │
│  ├── Direct (DCUtR hole-punching)                   │
│  ├── Community relays (DHT-discovered)              │
│  └── Official relays (goamp-relay-{eu,ru,us})       │
└─────────────────────────────────────────────────────┘
```

### Core entities

- **Account** — master ed25519 keypair. Only ever materialized in memory briefly during manifest signing; persisted only as BIP39 seed phrase written down by the user.
- **Device** — a physical install of GOAMP. Holds a sub-key (ed25519) signed by master + device metadata (name, OS, version, capabilities).
- **Device Manifest** — JSON document listing active sub-keys for an account, signed by master. Public, stored on relay and locally. Includes version (monotonic) + revoke list.
- **State Blob** — encrypted Automerge document containing user state. Stored on relay as opaque ciphertext.
- **Session** — ephemeral playback state (active device, current track, position, queue, flags). Not persisted on relay; propagated via pubsub.

### Trust separation

Relay sees: `account_pub_key` (for routing), `sub_key` of sender/recipient (for auth), timestamps, blob sizes, connection metadata.
Relay does NOT see: playlists, listening history, track IDs, settings, session content, remote commands — all are encrypted with keys derived from the master the relay does not hold.

## 5. Identity & Key Hierarchy

### Master key

- ed25519, generated by secure RNG on first run.
- Encoded as BIP39 12-word mnemonic (128 bits of entropy).
- Shown once to the user with a mandatory verification step ("enter words 3, 7, 11 to confirm you wrote them down").
- Never persisted to disk after this. Derived back from the seed phrase only when needed to sign a new manifest version (add device / revoke device / rotate).

### Sub-keys

- ed25519, generated per device.
- Signed by master via `sig = Sign(master_sk, "goamp-device-v1" || device_id || sub_pub || created_at)`.
- Persisted to OS keychain (macOS Keychain / Windows Credential Manager / Linux libsecret) on the device that owns it.
- Used for all day-to-day operations: signing relay requests, signing remote commands, deriving pubsub encryption keys, authoring Automerge ops.

### Device Manifest format

```json
{
  "version": 3,
  "account_pub": "ed25519:...",
  "devices": [
    {
      "sub_pub": "ed25519:...",
      "name": "Mac",
      "os": "darwin",
      "added_at": "2026-04-20T10:00:00Z",
      "master_sig": "..."
    }
  ],
  "revoked": [
    { "sub_pub": "ed25519:...", "revoked_at": "...", "reason": "lost" }
  ],
  "created_at": "2026-04-20T10:00:00Z",
  "master_sig": "<signature over the entire manifest body>"
}
```

- `version` is monotonically increasing. Relay accepts only versions strictly greater than its current cached value for the account.
- Anyone can verify a manifest using `account_pub`. Relay does this on every signed request and re-verifies on manifest update.

### State encryption key derivation

- `state_key = HKDF-SHA256(ikm=master_sk_seed, salt="goamp-v1", info="goamp-state-v1")`.
- Provisioned on a device at pairing time:
  - **QR pairing**: old device derives `state_key` from master (held in memory briefly during pairing) and transmits it to the new device over the authenticated libp2p pairing session. New device never sees the master.
  - **Seed recovery**: new device derives `state_key` directly from the entered seed.
- Cached in OS keychain alongside `sub_sk`.
- Unchanged on revoke by default. Paranoid mode triggers a rotation: new `state_key_v2` = HKDF with `info="goamp-state-v2"`, all existing state re-encrypted and pushed, old blob marked expired. Paranoid mode requires seed phrase entry.

## 6. Device Pairing & Lifecycle

### First-run / new account

1. Generate master keypair. Show BIP39 mnemonic with mandatory verification quiz.
2. Generate device sub-key. Sign with master.
3. Create Device Manifest v1 with this single device. Sign with master.
4. Wipe master from memory.
5. Push manifest to official relay. Cache locally.
6. Begin normal operation.

### Pair a new device (existing account, old device online)

1. New device: "Add to existing account" → generates candidate sub-key → shows QR code containing `goamp://pair?sub_pub=X&session=Y&nonce=N&relay_hint=Z`.
2. Old device: scans QR → opens secure libp2p session to new device (session Y) → prompts user for seed phrase.
3. User enters seed → old device derives master → signs new sub-key → builds manifest v+1 → signs manifest → wipes master.
4. Old device pushes new manifest to relay AND sends it directly to new device over the pairing session, along with the HKDF-derived state_key.
5. New device verifies manifest, stores state_key in keychain, pulls latest state blob from relay, decrypts, builds local Automerge doc.
6. New device joins pubsub topics; appears in "Devices" UI of all other devices.

### Recover on a new device (no old device available)

1. New device: "Recover from seed phrase" → 12-word input.
2. Derives master → fetches current manifest from official relay (public, any new device can fetch given account_pub, which is derived from master).
3. Generates new sub-key → signs new manifest version with all existing devices preserved, adding self → pushes → wipes master.
4. Derives state_key from master → fetches state blob → decrypts → joins pubsub.

### Revoke a device

1. On any active device: Settings → Devices → "Remove Phone" → prompts for seed phrase (master required).
2. Derives master → signs new manifest with the revoked sub_pub moved to `revoked` list → pushes → wipes master.
3. Relay invalidates its manifest cache. Next time the revoked device makes a signed request, it fails with 401 `revoked`.
4. Revoked device UI shows: "This device has been removed from your account. To rejoin, use Recover from seed phrase."
5. Optional paranoid mode: trigger state_key rotation (see 5.4).

### UX principles

- Seed phrase entry is required ONLY for manifest-modifying operations (add/revoke/rotate). Day-to-day usage uses cached sub-key.
- All manifest operations are atomic per-version. Relay rejects stale versions.
- Manifest updates are synchronized in-band: new manifest arrives before/with state changes that require new trust state.

## 7. User State Sync (CRDT)

### UserState schema (Automerge)

```
UserState {
  playlists: Map<playlist_id, Playlist>
    Playlist { name, tracks[], created_at, updated_at }
  liked_tracks: Set<track_id>
  disliked_tracks: Set<track_id>
  history: List<PlayEvent>         // only if history_sync is globally on
    PlayEvent { track_id, device_id, started_at, duration_listened, completed }
  subscriptions: Set<radio_source_id>
  settings: Map<key, value>         // cross-device settings only (skins, theme, hotkeys)
  taste_profile: TasteVector        // weights for recommendation engine
}
```

### Device-local (NOT synced)

- Volume
- Audio output device selection
- Window position/size
- Local cache index (which tracks are downloaded on this device)
- History from this device (when per-device contribution is off)

### Transport

- GossipSub topic: `/goamp/user-state/<account_pub_hash>`.
- Each Automerge operation is encrypted with `state_key` (ChaCha20-Poly1305, per-message nonce) and published to the topic.
- All online devices subscribe. Deltas propagate within ~50–500 ms when direct or via relay.
- Offline devices catch up on reconnect via snapshot + delta stream.

### Snapshot cadence

1. Realtime deltas: every Automerge op, immediate publish.
2. Snapshot push to relay:
   - Debounced: every 5 minutes if there are changes.
   - Immediate on significant events: playlist create/delete, device add/remove, app going to background (mobile), app quit, network loss (flush-on-exit).
   - Rate-limited: minimum 30 seconds between snapshots per device.
3. Reconnect catch-up: pull latest snapshot + deltas newer than local seen timestamp.
4. Compaction: weekly (keeps blob size bounded).

### Privacy switches

1. **Global** (account-level, in shared UserState): `history_sync: bool` — default `true`. Controls whether any PlayEvent is written into the shared Automerge history list.
2. **Per-device** (device-local setting): `this_device_contributes_history: bool` — default `true`. When `false`, PlayEvents from this device go only to local-only history and are never merged into shared history.
3. **Incognito session** (runtime flag): equivalent to setting per-device to `false` until app quit. One-click toggle in the UI.

### Conflict resolution

- Automerge handles concurrent edits automatically (CRDT semantics: Lamport timestamps + tombstones for list deletes).
- Like / dislike on the same track from two devices concurrently: last-write-wins by Automerge internal timestamps.
- Playlist order edits: merged position-preserving.

### Size estimate

Typical active user after 1 year: ~1–5 MB Automerge doc. Weekly compaction keeps this bounded.

## 8. Session & Remote Control

### Session schema

```
Session {
  active_device_id: sub_pub
  current_track: TrackRef { track_id, source, metadata }
  position_ms: u64
  position_updated_at: timestamp
  playback_state: Playing | Paused | Buffering | Stopped
  queue: List<TrackRef>
  queue_position: u32
  shuffle: bool
  repeat: Off | One | All
  last_heartbeat: timestamp
  session_version: u64      // Lamport clock for conflict resolution
}
```

- In-memory on the active device.
- Streamed to online devices via encrypted pubsub topic `/goamp/session/<account_pub_hash>`.
- Not persisted on relay (ephemeral by design).

### Active device state machine

```
  Idle --[play]--> Active
  Active --[pause/stop]--> Active (playback_state=Paused/Stopped)
  Active --[no heartbeat 60s (as observed by peers)]--> Stale
  Active --[takeover accepted]--> Idle
  Stale --[heartbeat]--> Active
  Stale --[takeover]--> Idle (no confirmation required; stale is fair game)
```

### Heartbeat

- Active device publishes session update: every 2 s while Playing, every 10 s while Paused/Stopped.
- Includes `position_ms` for smooth UI interpolation on controllers.

### Remote commands

```
Command {
  cmd: Play | Pause | Seek(ms) | Next | Prev | SetVolume(n)
     | AddToQueue(track) | RemoveFromQueue(idx) | SetShuffle(b) | SetRepeat(m)
     | PlayTrack(track)                 // immediate replace
     | Takeover { from_position: Option<u64> }
  issued_by: sub_pub
  issued_at: timestamp
  nonce: [u8; 16]
  signature: sig(issued_by sub_key)
}
```

- Transport: libp2p stream protocol `/goamp/remote/1.0`, attempted in cascade (direct → community relay → official relay).
- Active device verifies signature against current manifest, applies command, publishes updated Session.

### TTL & replay protection

- Commands with `issued_at` older than 30 seconds are rejected.
- Sliding-window nonce cache (30 s) on active device; duplicate nonces rejected.
- Controllers buffer commands locally while offline and re-issue only those still within TTL upon reconnect.

### Takeover flow

1. Controller taps "Play here" while another device is Active.
2. Dialog: "Mac is playing 'X'. [Transfer here] / [Control from here]".
3. Transfer: controller sends `Takeover { from_position: Some(current_position) }` to active device.
4. Active device pauses local playback, publishes Session with new `active_device_id`, bumps `session_version`.
5. New active device loads the same track at the given position, begins playback, starts heartbeat.
6. Force fallback: if previous active does not ACK within 5 s, controller becomes active anyway; pubsub merges by Lamport (`session_version`).

### Multi-controller

- Any number of non-active devices can send commands simultaneously.
- Active device applies in arrival order.
- Conflicting commands within the same second (e.g. play vs pause): last `issued_at` wins.

### Devices panel UI

- Lists all devices from manifest with presence + session role (Active / Controller / Offline).
- For Active: shows current track, progress bar, remote transport controls.
- For Offline: shows last-seen timestamp.

## 9. Relay & Transport

### Transport cascade (per connection attempt)

```
1. Direct libp2p         (DCUtR hole-punching, QUIC/TCP + Noise)
2. Community relay       (circuit-relay v2 via DHT-discovered peer)
3. Official relay        (hardcoded endpoints + DNS SRV fallback)
```

- Client races (1) and (3) in parallel; first success wins.
- (2) is attempted as mid-tier fallback when (1) fails and (3) is slow/unavailable.

### Official relay services

- **State Blob Storage**: PUT/GET encrypted snapshots keyed by `account_pub_hash`. Append-only within a retention window (keeps last 3 snapshots + delta log).
- **Device Manifest Storage**: signed-by-master, publicly fetchable by `account_pub`.
- **PubSub Relay**: forwards encrypted GossipSub messages on `/goamp/user-state/*` and `/goamp/session/*` topics between an account's devices.
- **Circuit Relay (libp2p)**: forwards realtime command streams `/goamp/remote/1.0`.
- **Presence Service**: reports which sub_keys of an account are currently online (used for Devices UI).

### Relay authorization

Each signed request includes `sub_pub` + signature. Relay verifies:
1. Signature is valid over the request body + timestamp.
2. `sub_pub` is present in the current manifest for `account_pub`.
3. `sub_pub` is not in the `revoked` list.

Manifest is cached on relay for 60 s; invalidated immediately on manifest update push.

### Discovery

- Official endpoints: compiled into the binary + `_goamp-relay._tcp.goamp.org` DNS SRV for failover.
- Community relays: advertised in libp2p DHT under `/goamp/relays`; client picks 3 by RTT.
- Geo-routing: client pings all official relays on startup, chooses lowest RTT.

### Rate limits

- State blob PUT: 1 per 30 s per account.
- PubSub publish: 100 msg/s per account.
- Manifest update: 10 per day per account.
- On exceed: HTTP 429; client exponential backoff.

### Infrastructure

- 3 official regions minimum for MVP: EU, RU, US.
- Stateless HTTP/WS front + shared S3-compatible storage for blobs + Redis for manifest cache.
- Primary region in RU jurisdiction (no PII stored — only ciphertext and pubkeys — so no ФЗ-152 processing obligations triggered).
- Community relay: documented setup + reference Docker image.

### Cost model (rough)

- State blobs: ~5 MB × 3 snapshots per user = 15 MB. 10k users ≈ 150 GB (negligible storage cost).
- PubSub throughput: ~1 msg/s/active-user.
- Main unknown: circuit-relay bandwidth for users behind symmetric NAT. Reevaluate after beta telemetry.

## 10. Error Handling

| Scenario | Behavior |
|---|---|
| All relays unreachable | Fall back to local-only. UI shows offline indicator. Changes buffer in local Automerge log, sync on reconnect. |
| Revoked device attempts to connect | Relay returns 401 `revoked`. Device shows: "This device was removed. Recover from seed phrase to rejoin." |
| Concurrent manifest updates (two devices revoke simultaneously) | Relay accepts only the higher version. Loser fetches latest, rebases, retries. |
| Seed phrase typo on recovery | BIP39 checksum catches most typos. Otherwise derived master won't match manifest signatures → "seed does not match account". |
| Active device crashes mid-track | 60 s heartbeat timeout → Session marked Stale. Controllers see "Mac offline — continue here?". |
| Corrupted state blob on relay | Client keeps last 3 snapshots locally; falls back to prior snapshot + replay log. Relay compaction ensures at least one valid snapshot is always available. |
| Clock skew vs TTL | Lamport timestamps for CRDT ordering; NTP best-effort for wall-clock. TTL uses relative `received_at − issued_at` with ±5 s tolerance. |
| Network partition — both devices think they are active | Pubsub merge by `session_version` Lamport clock; higher wins. Loser transitions to Controller role, shows notification. |

## 11. Security

- Master key in memory only during manifest signing; wiped immediately after.
- Seed phrase never written to logs, telemetry, or crash reports. Telemetry library has an explicit denylist.
- State blob: ChaCha20-Poly1305 AEAD with fresh nonce per snapshot, `account_pub_key` as associated data.
- PubSub messages: per-message ChaCha20-Poly1305 with fresh nonce.
- Forward secrecy: libp2p Noise handshake with ephemeral keys per transport session (standard libp2p behavior).
- Command replay protection: nonce + `issued_at` + 30 s sliding window nonce cache.
- Revocation in default mode blocks the revoked device's *access* to new data: relay rejects its requests (sub_pub in manifest's `revoked` list), and healthy peers refuse its signed pubsub messages for the same reason. The revoked device retains `state_key` in its keychain and could still decrypt any ciphertext it had already cached or obtains through a compromised/legacy relay. To guarantee cryptographic revocation, enable paranoid mode: it rotates `state_key` and re-encrypts all state so data published after revoke is unreadable with the old key. This is a deliberate trade-off: default mode is non-disruptive for other devices; paranoid mode forces every remaining device to re-fetch state with the new key.
- Relay compromise: maximum leak is metadata (account pubkey, online times, blob sizes). All content stays confidential.

## 12. Testing Strategy

### Unit tests (`goamp-node`, Go)

- Automerge merge scenarios (concurrent edits, offline merge, list reorder).
- Manifest signing/verification; add/revoke operations.
- Command TTL; nonce replay protection sliding window.
- State blob encrypt/decrypt round-trip.
- Sub-key chain validation.

### Integration tests (`goamp-node` + mock relay)

- End-to-end pairing flow: QR pair → manifest update → state sync.
- Recovery from seed phrase (with and without previously active devices online).
- Revocation applied; revoked device rejected by relay.
- Takeover flow with two `goamp-node` processes on localhost.

### E2E tests (2+ real Tauri clients + real relay)

- Two desktops + simulated phone: play on A → takeover from B → continue from position.
- Offline edit of the same playlist on two devices → reconnect → merged correctly.
- Revoke device → revoked client receives 401 and surfaces recovery UI.

### Chaos tests (using existing `testutil.NewNetwork`)

- Injected latency, partition, packet loss.
- Scenarios: "relay down, direct up", "direct down, relay up", "both down".

### Security tests

- Fuzz the manifest parser (malicious signatures, version rollback, malformed JSON).
- Replay attack: resend captured old commands → must be rejected.
- Tampered state blob: decryption fails → client falls back to local snapshot.

### Observability

- Relay metrics: connections, pubsub throughput, blob ops, revoke rate per account.
- Client telemetry (opt-in): pairing success rate, transport cascade depth distribution, takeover counts, offline-merge counts.
- Health topic: `/goamp/relay-health` pubsub for external monitoring.

## 13. Open Items (for implementation plan stage)

- Concrete library choice: Automerge-rs vs yrs (Yjs Rust port). Criteria: binary size, Go FFI maturity for the sidecar, snapshot size.
- Exact QR pairing wire protocol (raw libp2p stream vs. a small signed JSON envelope).
- Presence heartbeat frequency tuning (balance between stale-detection speed and battery on mobile).
- Relay admission control: should we require a proof-of-work or invite token at account creation to prevent free-storage abuse? Defer until abuse is observed.
- Mobile background behavior (iOS/Android) — defer with Phase 1 mobile scope.
